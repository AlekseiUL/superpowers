# Skill: Project Estimator - Сметчик проектов

## Описание
Этот навык предназначен для агента-сметчика, который будет оценивать стоимость, сроки и необходимые ресурсы для IT-проектов. Он работает на основе декомпозиции задач, эвристической оценки трудозатрат и рыночного бенчмаркинга, формируя обоснованное коммерческое предложение.

## Когда использовать
Используйте этот навык, когда необходимо:
- Оценить стоимость автоматизации, разработки нового продукта или фичи.
- Определить трудозатраты специалистов различных ролей.
- Обосновать предложенную стоимость для клиента.
- Сформировать коммерческое предложение с детальным описанием работ и бюджета.

## Рабочий процесс (Workflow)

Процесс оценки проекта состоит из следующих ключевых шагов:

### Шаг 1: Сбор и уточнение требований (Использует `superpowers:business-analyst`)

**Цель:** Преобразовать высокоуровневый запрос пользователя в структурированный список функциональных и нефункциональных требований, пользовательских историй (User Stories) и ключевых фич.

**Действия:**
1.  **Запрос:** Получите от пользователя описание проекта/автоматизации.
2.  **Анализ `superpowers:business-analyst`:** Делегируйте `superpowers:business-analyst` для анализа запроса, выявления неясностей, постановки уточняющих вопросов пользователю и формирования списка требований.
3.  **Результат:** Получите четко сформулированный список фич и их приоритетов.

### Шаг 2: Оценка технической сложности и архитектуры (Использует `role-api-architect`)

**Цель:** Определить технологический стек, общую архитектуру решения и оценить основные технические вызовы.

**Действия:**
1.  **Запрос:** Передайте список фич `role-api-architect`.
2.  **Анализ `role-api-architect`:** Делегируйте `role-api-architect` для предложения оптимальной архитектуры (микросервисы, монолит, serverless и т.д.), определения основных интеграций, выявления потенциальных технических сложностей и формирования высокоуровневого технического плана.
3.  **Результат:** Получите описание архитектуры и список ключевых технических компонентов.

### Шаг 3: Декомпозиция задач (Work Breakdown Structure - WBS)

**Цель:** Разбить каждую фичу на детальные, атомарные технические задачи, которые можно оценить по трудозатратам.

**Действия:**
1.  **Input:** Список фич (от `business-analyst`) и архитектурный план (от `role-api-architect`).
2.  **Процесс:**
    *   Для каждой фичи сгенерируйте список подзадач, относящихся к разным ролям (Backend, Frontend, QA, DevOps, Project Management, UI/UX).
    *   Примеры подзадач:
        *   Backend: Разработка API-эндпоинта, логика обработки данных, интеграция с БД, юнит-тесты.
        *   Frontend: Разработка компонента UI, интеграция с API, валидация форм, UI-тесты.
        *   QA: Написание тест-кейсов, ручное тестирование, автоматизация тестов.
        *   DevOps: Настройка CI/CD, развертывание на сервере, мониторинг.
3.  **Результат:** Детализированный список задач для каждой роли.

### Шаг 4: Эвристическая оценка трудозатрат (Effort Estimation)

**Цель:** Оценить время в человеко-часах (Min/Max/Likely) для каждой атомарной задачи.

**Действия:**
1.  **Input:** Детализированный список задач.
2.  **Процесс (эвристики для LLM):**
    *   Для каждой задачи примените следующую эвристику (внутренний промпт для LLM):
        > "Оцени трудозатраты (в человеко-часах) для задачи '[Название задачи]' с учетом '[Технологический стек]' и '[Сложность по архитектурному плану]' в трех значениях: T_optimistic (если все идет идеально), T_pessimistic (если все идет плохо) и T_likely (наиболее вероятное). Учитывай, что это будет выполнять [Роль специалиста]."
    *   Используйте формулу PERT для расчета ожидаемого времени:
        $$T_{expected} = \frac{T_{optimistic} + 4 \times T_{likely} + T_{pessimistic}}{6}$$
    *   Добавьте коэффициенты риска (RiskFactor) для сложных или неопределенных задач. Например:
        *   Стандартная задача: `RiskFactor = 1.0`
        *   Новая технология/сложная интеграция: `RiskFactor = 1.2 - 1.5`
        *   R&D/Высокая неопределенность: `RiskFactor = 1.5 - 2.0`
3.  **Результат:** Список задач с ожидаемыми трудозатратами в часах и примененными коэффициентами риска.

### Шаг 5: Финансовое моделирование

**Цель:** Преобразовать трудозатраты в финансовую оценку, используя тарифную сетку (Rate Card).

**Действия:**
1.  **Input:** Список задач с трудозатратами и ролями, а также `Rate Card` (внутренний параметр).
    *   Пример `Rate Card`:
        ```
        {
          "Backend Developer": {"Junior": 30, "Middle": 50, "Senior": 70, "Lead": 90},
          "Frontend Developer": {"Junior": 30, "Middle": 50, "Senior": 70, "Lead": 90},
          "QA Engineer": {"Junior": 25, "Middle": 40, "Senior": 60},
          "DevOps Engineer": {"Middle": 60, "Senior": 85},
          "UI/UX Designer": {"Middle": 55, "Senior": 75},
          "Project Manager": {"Senior": 80}
        }
        ```
2.  **Процесс:**
    *   Для каждой задачи определите требуемую роль и уровень специалиста.
    *   Примените соответствующую почасовую ставку.
    *   Рассчитайте стоимость каждой задачи: `Cost_task = T_expected_task * Rate_role * RiskFactor`.
    *   Суммируйте все `Cost_task` для получения `Total_Cost_Before_Margin`.
    *   Примените маржу предприятия (внутренний параметр, например, 20-30%) для получения `Final_Commercial_Offer`.
3.  **Результат:** Детализированная смета с разбивкой по задачам, ролям, часам и стоимости, а также общая сумма.

### Шаг 6: Бенчмаркинг и рыночная проверка (Использует `perplexity-api-expert` / `librarian`)

**Цель:** Сравнить полученную оценку с рыночными данными для аналогичных проектов.

**Действия:**
1.  **Запрос:** Используйте ключевые слова проекта (например, "стоимость разработки e-commerce платформы", "трудозатраты на интеграцию платежного шлюза") для поиска в интернете.
2.  **Анализ `perplexity-api-expert` / `librarian`:** Делегируйте `perplexity-api-expert` или `librarian` для сбора информации о рыночных ценах и трудозатратах по похожим проектам.
3.  **Процесс:** Сравните `Final_Commercial_Offer` с собранными рыночными бенчмарками.
4.  **Результат:** Отчет о сравнении и комментарии, если есть существенные отклонения. Если отклонение > 30%, требуется дополнительное обоснование.

### Шаг 7: Генерация коммерческого предложения и обоснования (Использует `copywriter`)

**Цель:** Сформировать финальный документ – коммерческое предложение (КП) – с детальным обоснованием стоимости.

**Действия:**
1.  **Input:** Вся собранная информация: список фич, архитектура, WBS, трудозатраты, финансовая оценка, рыночные бенчмарки, результаты сравнения.
2.  **Процесс:**
    *   Сгенерируйте вводную часть КП (описание проекта, цели).
    *   Включите детализированную смету.
    *   Сгенерируйте раздел "Обоснование стоимости", где объясняются основные факторы, влияющие на цену (сложность, риски, технологии, объем работ).
    *   Добавьте раздел "Сроки выполнения".
    *   Включите выводы по бенчмаркингу.
    *   Используйте `copywriter` для стилистической доработки и формирования убедительного текста.
3.  **Результат:** Готовое коммерческое предложение в удобном формате (например, markdown или PDF).

## Обязательные действия для агента, использующего этот навык:
1.  Всегда начинать с **Шага 1: Сбор и уточнение требований**.
2.  Четко фиксировать все входные данные и промежуточные результаты.
3.  Автоматически использовать требуемые навыки (`superpowers:business-analyst`, `role-api-architect`, `perplexity-api-expert` / `librarian`, `copywriter`) на соответствующих шагах.
4.  В случае существенного отклонения оценки от рыночных бенчмарков, обязательно указать это в обосновании.

## Запрещенные действия:
1.  Предоставлять окончательную смету без прохождения всех шагов рабочего процесса.
2.  Игнорировать факторы риска или не использовать формулу PERT для оценки времени.
3.  Не обосновывать финальную стоимость.
